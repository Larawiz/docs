# Primary Key

Larawiz automatically creates the id as primary column when using Quick Models, so there is no need to declare it.

    models:
  Post:
    title: string
    excerpt: string
    body: longText

    ¬†






    Schema::create('podcasts', function (Blueprint $table) {
    $table->id(); // Added automatically
    $table->string('title');
    $table->string('slug');
    $table->longText('body');
    $table->timestamps(); 
});

    You can change the name of the primary key using the id column followed by the new name. Larawiz will automatically point the primary key column name inside the model.



      ¬†




      models:
  Post:
    id: publication_number
    title: string
    excerpt: string
    body: longText


      ¬†




      class Post extends Model
{
    protected $primaryKey = 'publication_number';
    
    // ...
}

      ¬†






      Schema::create('podcasts', function (Blueprint $table) {
    $table->id('publication_number');
    $table->string('title');
    $table->string('slug');
    $table->longText('body');
    $table->timestamps(); 
});

      ID is just better

      Doing id: name instead of name: id ensures the model only has one primary key.

      # UUID as Primary Key

      When using Quick Models, you can exchange the default id column for a primary uuid column by just setting it. You can also change the default name, if you want, otherwise it will be uuid as default.

      For both Quick Models and Custom Models, Larawiz will automatically point and update the primary key inside the model, and include the HasUuidPrimaryKey trait for free alongside it, right out of the box!

      Do not add free traits

      You can use the option --no-free-traits when calling Larawiz to not include the HasUuidPrimaryKey and other free traits.



        ¬†




        models:
  Post:
    uuid: ~
    title: string
    excerpt: string
    body: longText




        ¬†
        ¬†
        ¬†




        class Post extends Model
{
    use HasUuidPrimaryKey;

    protected $primaryKey = 'uuid';
    protected $keyType = 'string';
    protected $incrementing = false;
    
    // ...
}

        <?php

namespace App\Models;

use Illuminate\Support\Str;

trait HasUuidPrimaryKey
{
    /**
     * Sets the primary key for the model as an UUID.
     *
     * @return void
     */
    protected function initializeHasUuidPrimaryKey()
    {
        $this->attributes[$this->getKeyName()] = $this->generateUuid();
    }

    /**
     * Generates an UUID to use as Primary Key.
     *
     * @return \Ramsey\Uuid\UuidInterface|string
     */
    protected function generateUuid()
    {
        return Str::uuid();
    }
}

        ¬†






        Schema::create('podcasts', function (Blueprint $table) {
    $table->uuid('uuid')->primary();
    $table->string('title');
    $table->string('slug');
    $table->longText('body');
    $table->timestamps(); 
});

        Polymorphic UUID

        Having uuid as primary column will automatically change morphs columns to uuidMorphs automatically when using the model on polymorphic relations.

        # Custom Primary Key

        When using Custom Models, Larawiz will hands-off the primary key declaration to you. You have three options:

        1. simply add an id or uuid manually,
        2. use another Primary Key,
        3. or just disable it altogether.

        No Composite Primary Keys

        Eloquent ORM doesn't support Composite Primary keys (opens new window) (made from multiple columns). For that reason, these are also no supported in Larawiz.

        # Adding the ID or UUID manually

        By default, if there is id, uuid, or an auto-incrementing column defined, it will be understood the model has a primary key. For example, we can do that with uuid.




          ¬†





          models:
  Podcast:
    columns:
      uuid: ~
      title: string
      excerpt: string
      body: longText
      timestamps: ~


          ¬†
          ¬†
          ¬†




          class Podcast extends Model
{
    protected $primary = 'uuid';
    protected $keyType = 'string';
    protected $incrementing = false;

    // ...
}

          ¬†






          Schema::create('podcasts', function (Blueprint $table) {
    $table->uuid('uuid')->primary();
    $table->string('title');
    $table->string('excerpt');
    $table->longText('body');
    $table->timestamps();
});

          # Using another column as Primary Key

          Alternatively, you can set any other column as primary by using the primary key. Larawiz will guess the rest based on the column you point as primary key, like the type and incrementing nature.









            ¬†

            models:
  Podcast:
    columns:
      title: string
      excerpt: string
      body: longText
      timestamps: ~

    primary: title


            ¬†
            ¬†
            ¬†




            class Podcast extends Model
{
    protected $primary = 'title';
    protected $keyType = 'string';
    protected $incrementing = false;

    // ...
}

            ¬†





            Schema::create('podcasts', function (Blueprint $table) {
    $table->string('title')->primary();
    $table->string('excerpt');
    $table->longText('body');
    $table->timestamps();
});

            If you need to manually set the primary key properties instead of letting Larawiz guess it for custom or non-standard columns, you can add type and incrementing values inside the primary key:







              ¬†
              ¬†
              ¬†
              ¬†

              models:
  Podcast:
    columns:
      foo: custom
      # ...

    primary:
      column: foo
      type: bar
      incrementing: false


              ¬†
              ¬†
              ¬†




              class Podcast extends Model
{
    protected $primary = 'foo';
    protected $keyType = 'bar';
    protected $incrementing = false;

    // ...
}

              ¬†




              Schema::create('podcasts', function (Blueprint $table) {
    $table->custom('foo')->primary();
    
    // ...
});

              # No primary key

              On the other hand, if you want your model to not have any primary key, simply don't issue an id, uuid, or auto-incrementing column at all.

              Always use Primary Keys

              Primary keys identify a single row in the database. Most of the time you will want one in your table.







                models:
  Podcast:
    columns:
      title: string
      slug: string


                ¬†
                ¬†




                class Podcast extends Model
{
    protected $primary = null;
    protected $incrementing = false;

    // ...
}

                Schema::create('podcasts', function (Blueprint $table) {
    $table->string('title');
    $table->string('slug');
});
